# Reference Artifacts

## Purpose

This document establishes **Reference Artifacts** as a foundational concept in KDSE. Reference Artifacts are the engineering evidence from which Engineering Knowledge is derived. Understanding Reference Artifacts is essential for understanding how KDSE transforms raw engineering information into authoritative Engineering Knowledge.

## What Is a Reference Artifact

A **Reference Artifact** is any existing source of engineering information. Reference Artifacts are engineering evidence.

A Reference Artifact may contain:

- Engineering concepts
- Implementation details
- Architecture
- Vendor-specific information
- Incomplete information
- Conflicting information

Reference Artifacts are **not** Engineering Knowledge. A Reference Artifact may contain knowledge, but the artifact itself is not authoritative Engineering Knowledge.

## Reference Artifacts Are Not Engineering Knowledge

A Reference Artifact shall never be considered authoritative Engineering Knowledge by itself. Engineering Knowledge must always be derived.

### Why Reference Artifacts Are Not Engineering Knowledge

Reference Artifacts lack the characteristics required for authority:

| Characteristic | Reference Artifact | Engineering Knowledge |
|---------------|-------------------|----------------------|
| Validation | Not validated | Validated against evidence |
| Structure | May be unstructured | Structured with dependencies |
| Independence | Implementation-dependent | Implementation-independent |
| Authority | No inherent authority | Authoritative by design |
| Provenance | May be unclear | Explicitly documented |

### The Transformation Required

Raw Reference Artifacts do not become Engineering Knowledge through collection alone. They must be:

1. **Analyzed** to identify engineering evidence
2. **Interpreted** to understand engineering intent
3. **Validated** to confirm accuracy
4. **Transformed** into implementation-independent statements

## Types of Reference Artifacts

Reference Artifacts may take many forms. Common types include:

### Project Documentation

- Requirements documents
- Design specifications
- System manuals
- Operation procedures
- Training materials

### Implementation Artifacts

- Source code
- Configuration files
- Build scripts
- Deployment scripts
- Database schemas

### Communication Artifacts

- Meeting notes
- Email discussions
- Chat transcripts
- Decision records
- Change requests

### Vendor Documentation

- Product manuals
- API documentation
- Integration guides
- Release notes
- Support tickets

### Standards and Specifications

- Industry standards
- Regulatory requirements
- Protocol specifications
- Interface definitions

### Commissioning Records

- Test results
- Acceptance records
- Commissioning reports
- Validation evidence

### Diagrams and Models

- System architecture diagrams
- Sequence diagrams
- State diagrams
- Data flow diagrams

## Reference Artifact Characteristics

### Characteristic 1: Origin

Reference Artifacts originate from sources outside the KDSE knowledge derivation process. They are:

- Written by domain experts
- Created during project execution
- Generated by vendors
- Produced by regulatory bodies
- Contributed by stakeholders

### Characteristic 2: Completeness

Reference Artifacts vary in completeness:

- Some are complete and authoritative
- Some have gaps requiring interpretation
- Some are fragments without full context
- Some contain contradictions

### Characteristic 3: Perspective

Reference Artifacts reflect the perspective of their creators:

- Vendors emphasize product capabilities
- Implementers focus on realized functionality
- Operators prioritize operational concerns
- Regulators emphasize compliance

### Characteristic 4: Currency

Reference Artifacts may be current or outdated:

- Some reflect current system state
- Some document historical configurations
- Some describe planned but unimplemented features
- Some describe superseded systems

## The Role of Reference Artifacts in KDSE

Reference Artifacts serve as the foundation for Engineering Knowledge derivation.

### Reference Artifacts Provide Evidence

Reference Artifacts provide the evidence from which Engineering Knowledge is derived. The evidence includes:

- **Facts**: Observable system properties
- **Assertions**: Claims about system behavior
- **Decisions**: Documented choices and their rationale
- **Constraints**: Stated limitations and requirements

### Reference Artifacts Require Analysis

Reference Artifacts do not speak for themselves. They require:

- **Contextualization**: Understanding the circumstances under which the artifact was created
- **Interpretation**: Extracting the engineering meaning from the artifact's content
- **Correlation**: Comparing multiple artifacts to validate or contradict findings
- **Validation**: Confirming that interpretations accurately represent the engineering intent

### Reference Artifacts Support but Do Not Replace Engineering Knowledge

Reference Artifacts support Engineering Knowledge. They do not replace it.

When multiple Reference Artifacts agree, Engineering Knowledge becomes stronger.

When Reference Artifacts disagree, the contradiction shall be preserved. Contradictions shall never be silently resolved.

## Reference Artifact Analysis

The process of analyzing Reference Artifacts involves:

### Step 1: Artifact Identification

Identify all Reference Artifacts relevant to the engineering domain:

- Search project documentation
- Examine existing implementation
- Review vendor materials
- Consult stakeholder sources

### Step 2: Artifact Classification

Classify each Reference Artifact by type:

- Project Documentation
- Implementation Artifact
- Communication Artifact
- Vendor Documentation
- Standard or Specification

### Step 3: Evidence Extraction

Extract engineering evidence from each artifact:

- Identify factual statements
- Note assertions about behavior
- Record documented decisions
- Capture stated constraints

### Step 4: Correlation

Correlate evidence across multiple artifacts:

- Identify agreements
- Identify contradictions
- Note gaps in coverage
- Assess completeness

### Step 5: Interpretation

Interpret the collected evidence:

- Determine engineering intent
- Assess implementation independence
- Validate against other evidence
- Formulate Engineering Knowledge statements

## Evidence Strength

Evidence Strength reflects how well Engineering Knowledge is supported by Reference Artifacts.

### Evidence Strength Levels

Evidence Strength is determined by independent supporting engineering evidence:

**★★★★★ - Strong Evidence**

Supported by multiple independent Reference Artifacts:

- Project Documentation
- Node-RED flows
- Vendor Manual

**★★★★☆ - Good Evidence**

Supported by project documentation and one additional source:

- Project Documentation
- Node-RED

**★★★☆☆ - Moderate Evidence**

Supported by project documentation only:

- Project Documentation

**★★☆☆☆ - Weak Evidence**

Supported by limited sources:

- Vendor Manual only
- Single implementation artifact only

**★☆☆☆☆ - Minimal Evidence**

Supported by indirect or inferential evidence:

- Inferred from implementation
- Derived from vendor documentation without project confirmation

### Evidence Strength and Authority

Evidence Strength affects confidence in Engineering Knowledge but does not determine authority. Authority derives from the structured knowledge process, not from the quantity of evidence.

Engineering Knowledge with weak evidence may still be authoritative if properly validated. Engineering Knowledge with strong evidence may still require additional validation.

## Handling Contradictions

When Reference Artifacts contradict each other:

### Step 1: Document the Contradiction

Preserve the contradiction in the analysis:

- Note what each artifact claims
- Identify the specific point of disagreement
- Record the source of each claim

### Step 2: Assess Engineering Impact

Evaluate the engineering significance:

- Does the contradiction affect system behavior?
- Does it impact safety, reliability, or performance?
- Does it relate to a critical interface or function?

### Step 3: Determine Resolution Path

Classify the contradiction for resolution:

- **Engineering Knowledge Question**: Cannot be resolved from available artifacts
- **Architecture Question**: Relates to software organization
- **Implementation Question**: Relates to implementation technology

### Step 4: Preserve the Contradiction

Do not silently resolve contradictions. The contradiction shall be preserved in the analysis and communicated to stakeholders.

Operator review is required only when contradictions affect Engineering Knowledge.

## Reference Artifacts and the KDSE Lifecycle

Reference Artifacts occupy a specific position in the KDSE lifecycle:

```
Reference Artifact
        ↓
Reference Analysis
        ↓
Engineering Knowledge Derivation
        ↓
Evidence Correlation
        ↓
Knowledge Validation
        ↓
Approved Engineering Knowledge
```

### Phase 1: Reference Artifact

Collect and organize Reference Artifacts from available sources.

### Phase 2: Reference Analysis

Analyze Reference Artifacts to identify engineering evidence.

### Phase 3: Engineering Knowledge Derivation

Derive Engineering Knowledge statements from the analyzed evidence.

### Phase 4: Evidence Correlation

Correlate evidence across multiple Reference Artifacts to strengthen or question derived knowledge.

### Phase 5: Knowledge Validation

Validate derived Engineering Knowledge through structured review.

### Phase 6: Approved Engineering Knowledge

Authorize validated Engineering Knowledge for use in architecture and implementation.

## Repository First Principle

Before asking the operator for clarification, analyze all available Reference Artifacts.

### Sources to Search

Search the following sources before requesting operator input:

1. Project documentation
2. Existing implementation
3. Diagrams
4. Configuration
5. Source code
6. Vendor documentation
7. Engineering calculations
8. Commissioning records
9. Existing knowledge artifacts

### If Sufficient Evidence Exists

If sufficient engineering evidence exists in Reference Artifacts, derive the Engineering Knowledge without asking the operator.

### If Evidence Is Insufficient

If Reference Artifacts are insufficient to derive Engineering Knowledge, classify the question and proceed according to Question Classification principles.

## Summary

Reference Artifacts are the engineering evidence from which Engineering Knowledge is derived. They are:

- **Sources of evidence, not authoritative knowledge**
- **Analyzed, not simply collected**
- **Correlated across multiple sources**
- **Interpreted for engineering intent**
- **Preserved in contradiction, not silently resolved**

Understanding Reference Artifacts is essential for understanding how KDSE transforms raw engineering information into authoritative, implementation-independent Engineering Knowledge.
