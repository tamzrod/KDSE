# Scope

## What KDSE Is

KDSE defines:

1. **Artifact Types**: The canonical set of software engineering artifacts (knowledge, architecture, implementation, verification, governance)

2. **Artifact Relationships**: How artifacts relate to each other, including derivation rules and consistency requirements

3. **Artifact Authority**: The hierarchy of artifact authority, establishing knowledge as the authoritative source

4. **Traceability Requirements**: The mandatory traceability paths between artifact types

5. **Engineering Model**: The lifecycle model governing artifact creation, evolution, and retirement

6. **Core Principles**: The timeless principles that guide all KDSE practice

## What KDSE Is Not

KDSE is not:

1. **A Knowledge Management System**: KDSE does not prescribe how to store, retrieve, or share knowledge. It defines what constitutes knowledge as an artifact and how knowledge relates to other artifacts.

2. **A Documentation Framework**: KDSE does not prescribe documentation formats, templates, or structures beyond artifact definitions. Documentation practices emerge from artifact definitions, not from KDSE itself.

3. **A Programming Methodology**: KDSE does not prescribe coding practices, language choices, or implementation patterns. Implementation derives from knowledge and architecture, not from KDSE principles.

4. **An AI Methodology**: KDSE does not prescribe AI tools, models, or integration patterns. AI systems may consume and produce KDSE artifacts, but KDSE remains AI-agnostic.

5. **A Testing Framework**: KDSE does not prescribe testing methodologies, frameworks, or tools. Verification artifacts derive from knowledge and architecture requirements.

6. **A Project Management Framework**: KDSE does not prescribe how to organize teams, manage timelines, or allocate resources. Governance artifacts may reference project management concerns, but KDSE itself remains neutral.

7. **A Domain-Specific Methodology**: KDSE does not presume expertise in any specific domain. Domain knowledge exists within the knowledge artifact type.

8. **A Tool or Platform**: KDSE does not prescribe or recommend specific tools, platforms, or technologies. Tooling decisions are implementation concerns.

## Non-Goals

KDSE explicitly does not pursue:

1. **Universal Adoption**: KDSE does not seek to replace all other software engineering methodologies. Teams may adopt KDSE alongside other practices.

2. **Certification Programs**: KDSE does not establish certification bodies or credential requirements.

3. **Tool Ecosystems**: KDSE does not develop or endorse specific tools.

4. **Industry Standards**: KDSE does not seek formal standardization through external bodies.

5. **Academic Research**: While KDSE may inform research, academic research is not KDSE's objective.

## Boundaries

KDSE is bounded by:

1. **Artifact-Centric**: KDSE addresses artifacts and their relationships. It does not address organizational structures, team dynamics, or individual competencies beyond artifact responsibilities.

2. **Technology-Neutral**: KDSE makes no assumptions about programming languages, frameworks, platforms, or infrastructure.

3. **Language-Independent**: KDSE makes no assumptions about natural languages used for knowledge artifacts.

4. **Industry-Agnostic**: KDSE applies to software engineering in any industry context.

5. **Scale-Agnostic**: KDSE applies to software engineering regardless of project scale, from individual developers to large organizations.

## Scope Creep Prevention

The following are explicitly excluded from KDSE:

- Coding standards and style guides
- Testing frameworks and methodologies
- CI/CD pipeline definitions
- Deployment strategies
- Infrastructure definitions
- Data modeling approaches
- API design patterns
- User interface guidelines
- Security frameworks
- Performance optimization techniques

These concerns may be addressed through domain-specific extensions built upon KDSE's foundation.
